# 작업 체크리스트 — 가이드 작성/검증 전 점검 18항목

가이드를 작성·수정·검증하기 전 이 체크리스트로 점검한다. 컨텍스트가 길어질수록 원칙을 잊기 쉬우므로 매 구현 사이클의 입구로 사용한다.

> **Tradeoff**: 18개 중 15개는 `/guide-audit`가 자동 점검한다. 사람이 매번 봐야 할 항목은 수동 점검 3개(16~18)뿐.

## 사용 순서

1. 변경 범위 한 줄 명시
2. 적용 범위 표에서 대상 파일 종류 확인
3. 영향 받는 항목만 점검 (통과 / 보정 / 보류)
4. 구현 → `/guide-audit` 회귀 검증

## 적용 범위 (어느 항목이 어느 파일에 적용되는가)

## 체크리스트 18항목

### A. 본질 — Non-obvious invariants only

1. 코드/`ls`로 알 수 있는 내용 없음 (디렉토리 트리·폴더 설명·기술 스택 나열·"이 프로젝트는 X 사용" 류 5번 이상 ❌)
2. HOW NOT 항목마다 이유 명시 (`항목 — 이유` 패턴)

### B. 안티패턴 회피

3. G1: 디렉토리 트리 (`├`/`└` 5줄 이상) 없음
4. G2: 인사말 / 마무리 멘트 없음
5. G3: DRY/SOLID/KISS 같은 일반 BP 3개 이상 없음

### C. 구체성 & 측정 가능성

6. 추상 표현 ("잘", "적절히") 최소화
7. 측정 가능한 규칙 (절대 경로·숫자 임계·MUST/NEVER 등) 5개 이상

### D. 구조 & 길이

8. 100줄 이하 (영역 가이드·root map만 적용)
9. 마크다운 구조화 (H1×1 + H2≥3)
10. HTML 주석으로 사람용 안내 격리 (컨텍스트 주입 전 제거됨 → 토큰 비용 0)

### E. 명령어 — 결과 품질 2~3배

11. 빌드/테스트/린트/타입체크 명령어 명시 (4 카테고리 중 ≥3개)
12. 명령어 가드 1개 이상 (watch 모드 금지, no-verify 금지 등)

### F. @import & 참조

13. @import 최소주의 — Both sync + 영역 진입 후 강결합만 허용 (자세한 룰: `rubric.md`)
14. 그 외 모든 참조는 마크다운 링크 (영역 라우팅·외부 docs·약결합)

### G. Living document

15. 실수 발생 시 `/learn`으로 즉시 반영하는 사이클 (LEARNED CAUTIONS 누적 항목 ≥1. 신규 생성 시 placeholder OK)

### H. 수동 점검 항목

16. 첫 줄에 명확한 목적 1문장 — 가이드가 무엇인지·왜 있는지 즉시 식별
17. Tradeoff/제약 명시 — 룰의 비용/한계 한 줄 (해당 시)
18. Surgical changes — 변경한 모든 라인이 요청 추적 가능. 인접 코드 임의 "개선" 금지
